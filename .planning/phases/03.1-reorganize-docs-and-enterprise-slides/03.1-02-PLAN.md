    ```
    <!-- _footer: "" -->

    # Branching Strategy for Enterprise Teams

    ### Git Flow vs. Trunk-Based Development — when to use each and how to migrate

    &nbsp;

    - This talk covers the **two dominant models** for feature-driven application teams
    - After this you will know which model fits your team and what migration looks like
    - Audience: teams delivering features, not OSS library maintainers
    ```

    Slide 2 — The Core Question:
    Title: "Which Branch is Always Releasable?"
    Content: a two-column comparison showing Git Flow answer (`main` — only after release merge) vs TBD answer (`main` / trunk — always). Use a table. Pull values from the Quick Reference table in enterprise/release-strategy.md rows: "Stable branch", "Deployment frequency", "DORA profile".

    Slide 3 — Git Flow Branch Structure:
    Title: "Git Flow: Five Branch Types"
    Content: the five-branch code block from section 1.1 of enterprise/release-strategy.md:
    ```
    main              ← production-only; every commit is a release tag
    develop           ← integration branch; "next release" in progress
    feature/ABC-123   ← individual feature work; branches off develop
    release/2.3.0     ← release stabilization; branches off develop
    hotfix/fix-null   ← emergency fix on production; branches off main
    ```
    Add 3 bullet points: (1) main and develop are permanent; (2) feature/release/hotfix are temporary; (3) main = released code only, develop = next release accumulation.

    Slide 4 — Git Flow Merge Ceremony:
    Title: "Git Flow: Four Merge Choreographies"
    Content: a table with four rows (Feature merge, Release merge, Hotfix merge, Back-merge) showing direction and description. Derive from section 1.3 of enterprise/release-strategy.md. Add one key observation bullet: "Every release requires merging in both directions (develop→release→main AND main→develop back-merge). Miss one and branches diverge."

    Slide 5 — Where Git Flow Breaks:
    Title: "Git Flow: Where It Breaks Down"
    Content: pull from section 1.4 ("Where Git Flow Breaks Down") of enterprise/release-strategy.md. List exactly three failure modes with concise one-line descriptions:
    - Long-lived feature branches → merge conflicts compound over weeks
    - Merge debt → develop drifts from main; back-merges become risky
    - CI/CD friction → release branch stabilization limits deploy frequency
    Speaker note: mention that these are not hypothetical — they are the specific pain points teams describe before migrating.

    Slide 6 — Trunk-Based Development Structure:
    Title: "Trunk-Based Development: One Direction"
    Content: the branch structure from section 2.1:
    ```
    main (trunk)        ← always releasable
    feature/short       ← max 1-2 days; merge before it diverges
    release/2.3.0       ← cut from trunk days before ship; cherry-pick fixes
    ```
    Bullet points: (1) No develop branch — trunk IS the integration surface; (2) Short-lived branches prevent merge debt; (3) Release branches are optional and lightweight.

    Slide 7 — Feature Flags: The Enabling Mechanism:
    Title: "Feature Flags: How TBD Ships Incomplete Features"
    Content: the four flag types from section 3.1 of enterprise/release-strategy.md (Fowler taxonomy). Use a table with columns: Type | Purpose | Lifespan. Rows: Release flag (hide incomplete work, days–weeks), Experiment/A-B flag (measure variants, days–weeks), Ops flag (circuit breaker, permanent), Permission flag (entitlements, permanent).
    Speaker note: "The key insight: TBD ships code continuously but controls feature exposure separately. The branch is gone in 24 hours; the flag controls whether users see the feature."

    Slide 8 — Flag Lifecycle and Cleanup:
    Title: "Flag Debt: The Hidden Cost"
    Content: three-stage flag lifecycle (dark → canary → GA → cleanup) from section 3.3. Add two bullet points: (1) Release flags MUST be cleaned up post-GA — they accumulate as technical debt; (2) Stale flags = dead code paths that still need test coverage and still risk runtime errors.

    Slide 9 — Parallel Features: Side-by-Side:
    Title: "Two Features in Flight: Git Flow vs. TBD"
    Content: use the scenario from section 4 or section 1.2/2.2 of enterprise/release-strategy.md showing Developer A (feature/user-auth) and Developer B (feature/payment-redesign) working in parallel. Show how Git Flow isolates by branch (merge conflict risk at integration) vs TBD isolates by flag (merge daily, flag hides unfinished work). Keep concise — two short code blocks or a side-by-side table.

    Slide 10 — Migration Path:
    Title: "Migrating from Git Flow to Trunk: Four Stages"
    Content: the four migration stages from section 4 of enterprise/release-strategy.md. Use a numbered list showing stage name and one-line description of what changes. Add the key prerequisite bullet: "CI must be fast (< 10 min) and reliable (< 2% flaky) before Stage 2."

    Slide 11 — Decision Guide:
    Title: "Which Model Fits Your Team?"
    Content: decision criteria table from section 5 of enterprise/release-strategy.md. Show at minimum: deployment frequency need, team CI maturity, feature flag tooling, release cadence. Two columns: "Git Flow fits when..." vs "TBD fits when...". Keep to 4-5 rows.

    Slide 12 — Summary (no page number):
    Title: "What You Now Know"
    Content: a summary table with three columns: Concept | Git Flow | TBD. Rows covering: stable branch, parallel features, release process, CI requirement, feature isolation mechanism.
    Footer references (same style as OSS slides closing slide):
    ```
    📄 Reference doc: `docs/enterprise/release-strategy.md`
    📊 Diagrams: `docs/enterprise/diagrams.md`
    ```
    Speaker note: suggest 3 common questions — (1) "When should we migrate?" (2) "Can we do partial TBD?" (3) "What if our CI is slow?"

    3. GENERAL RULES:
    - Every slide except the title and summary uses `paginate: true` (inherited from front matter — no per-slide override needed)
    - Add `<!-- SPEAKER NOTE: ... -->` comments for slides 2, 5, 7, 10, 12
    - Use `&nbsp;` blank lines to add vertical breathing room between sections (as in OSS slides.md)
    - Code blocks use triple-backtick fences, no language specifier unless it's bash/xml
    - Keep each slide to content that fits a single screen — do not pack more than ~8 lines of prose or table rows per slide
