# Enterprise Version Bump and Release Notes

**Audience:** Mid-level developers on enterprise teams managing versioned releases in Git Flow or trunk-based development
**Last updated:** 2026-04-25

---

## Quick Reference

| Dimension | Git Flow | Trunk-Based Development |
|---|---|---|
| **Version bump timing** | On release branch cut — immediately after `git checkout -b release/x.y.z`, before any stabilization work | At tag time — manual bump commit on trunk just before tagging, or automated by CI (release-please / semantic-release) |
| **Who makes the bump commit** | Release manager (or a CI job triggered by the branch cut) | Developer (manual) or CI pipeline (automated) |
| **Where bump lives in branch graph** | On the `release/x.y.z` branch; merged to `main` and back to `develop` | On trunk (`main`); no separate release branch in the normal flow |
| **Release notes authoring** | On the release branch after stabilization, before merge to `main` | From trunk commits since the last tag; generated by tooling (release-please PR or semantic-release CI job) or manually |
| **Release notes committed to** | Release branch → merges to `main` alongside the code | `main` — via a merged Release PR (release-please) or a direct CI commit (semantic-release) |
| **Tooling** | git-cliff (run on release branch), manual version bump, commitlint | release-please (PR-based), semantic-release (fully automated), git-cliff (changelog only) |

---

## 1. Version Bump Timing and Location

### 1.1 Git Flow

In Git Flow the version bump commit is made on the release branch **immediately after the branch is cut from `develop`**, before any stabilization or bug-fix work begins. This gives every commit on the release branch a deterministic version identity and avoids a situation where the version number is unknown during QA.

**Sequence:**

```bash
# 1. Cut the release branch from develop
git checkout develop
git pull origin develop
git checkout -b release/2.4.0

# 2. Bump the version file — only change is the version string
# (e.g., package.json, pom.xml, pyproject.toml)
npm version 2.4.0 --no-git-tag-version   # or edit pom.xml / pyproject.toml manually

# 3. Commit the bump immediately
git add package.json
git commit -m "chore(release): bump version to 2.4.0"

# 4. Stabilization fixes are made on this branch from here
```

**Merge-back requirement:** After the release branch merges to `main` and is tagged, it must also merge back to `develop`. This merge-back must carry the version bump. Without it, `develop` would still carry the old version string (e.g., `2.3.0`) even after `2.4.0` ships. The convention is to immediately bump `develop` to the next snapshot or pre-release version after the merge-back:

```bash
# After release/2.4.0 has merged to main and been tagged v2.4.0:
git checkout develop
git merge release/2.4.0

# Bump develop to the next development cycle version
# (e.g., 2.4.1-SNAPSHOT in Maven, or 2.5.0-alpha.0 in semver packages)
git add package.json
git commit -m "chore(release): bump develop to 2.4.1-SNAPSHOT"
```

### 1.2 Trunk-Based Development

In trunk-based development there is no release branch cut moment in the normal flow — trunk is always releasable. This means the version bump happens **at or just before tag time**, either via a manual commit on trunk or fully automated by CI.

**Sub-pattern A: Tag-time manual bump**

The developer bumps the version file on trunk directly and tags immediately after:

```bash
# 1. Bump on trunk
git checkout main
git pull origin main
npm version 2.4.0 --no-git-tag-version
git add package.json
git commit -m "chore(release): bump version to 2.4.0"

# 2. Tag
git tag v2.4.0
git push origin main --tags

# 3. CI picks up the tag and builds the release artifact
```

**Sub-pattern B: Automated CI-driven bump**

CI detects a release trigger (a merged PR, a label, or a commit on `main` that matches a conventional commits threshold) and performs the bump without human intervention:

```yaml
# .github/workflows/release.yml
on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: google-github-actions/release-please-action@v4
        with:
          release-type: node
          manifest-file: .release-please-manifest.json
```

In this pattern: commits land on `main` → release-please opens or updates a Release PR with the version bump and CHANGELOG diff → team merges the PR → action tags the release and creates the GitHub Release. The bump commit and tag are created automatically; no human makes the bump commit.

## 2. Release Notes and Changelog Authoring

### 2.1 Git Flow

Release notes are authored **on the release branch**, typically as a `CHANGELOG.md` commit, after stabilization is complete and before the final merge to `main`. The release manager (or tooling) inspects commits from the previous release tag to HEAD on the release branch, groups them by conventional commit type, and writes or generates the entry.

**Sample CHANGELOG.md entry format:**

```markdown
## [2.4.0] - 2026-04-25

### Added

- feat(auth): support PKCE flow for OAuth2 clients (#412)
- feat(api): add pagination to /users endpoint (#398)

### Fixed

- fix(db): prevent connection pool exhaustion under high load (#407)
- fix(auth): handle expired refresh tokens gracefully (#401)

### Breaking Changes

- feat!(config): remove deprecated `AUTH_SECRET` env var — use `AUTH_TOKEN_SECRET` (#389)
```

The changelog commit is part of the release branch and merges to `main` alongside the code, so `main`'s history contains the release notes for every version.

### 2.2 Trunk-Based Development

Release notes are generated from commits on trunk since the previous release tag. The tooling choice determines whether a human reviews the diff before it is committed:

**Using release-please (PR-based):**
The tool opens a "Release PR" (titled `chore(main): release X.Y.Z`) that contains the auto-generated `CHANGELOG.md` diff alongside the version bump. The team reviews it and merges when ready. Merging = committing the changelog, bumping the version, and creating the GitHub Release — all in one atomic operation.

**Using semantic-release (fully automated):**
No PR is involved. On every push to `main`, the tool analyzes commits since the last tag, determines the next version using conventional commits rules, writes `CHANGELOG.md`, commits it, creates the tag, and publishes the GitHub Release — all in a single CI job. No human step after the triggering merge.

**Manual (no tooling):**
Same shape as Git Flow: inspect `git log v2.3.0..HEAD`, group commits by type, write a `CHANGELOG.md` entry, commit to trunk, tag.

## 3. Tooling Patterns

### 3.1 Conventional Commits

Conventional Commits is the commit message convention that all three tools below require as input. It gives commit messages a parseable structure so tools can determine version bump magnitude and generate changelogs automatically.

**Format:**

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
```

**Standard types and their version bump impact:**

| Type | Description | Version bump |
|------|-------------|--------------|
| `feat` | New feature | Minor (1.x.0 → 1.x+1.0) |
| `fix` | Bug fix | Patch (1.x.y → 1.x.y+1) |
| `chore` | Build process, tooling | None |
| `docs` | Documentation only | None |
| `style` | Formatting, whitespace | None |
| `refactor` | Code restructure, no behavior change | None |
| `test` | Adding or correcting tests | None |
| `perf` | Performance improvement | None |

**Breaking change notation — two forms:**

1. `!` after the type/scope:
   ```
   feat!: drop Node 14 support
   ```

2. `BREAKING CHANGE:` footer:
   ```
   feat(auth): migrate to OAuth2 only

   BREAKING CHANGE: Basic Auth endpoints removed. Clients must use /oauth/token.
   ```

Both forms trigger a **major** version bump (x.0.0).

**Complete commit message examples:**

```
feat(api): add cursor-based pagination to /orders endpoint

Replaces offset pagination. Clients should migrate to ?cursor= parameter.
Offset parameters still accepted but deprecated.
```

```
fix(db): prevent connection leak when transaction rollback fails

Wrapped rollback in finally block so the connection returns to pool
regardless of rollback success.
```

```
feat!(config): remove deprecated AUTH_SECRET environment variable

BREAKING CHANGE: AUTH_SECRET is no longer read. Set AUTH_TOKEN_SECRET instead.
All existing deployments must update their environment configuration before
upgrading to this version.
```

**Enforcement tooling:**

- `commitlint` with `@commitlint/config-conventional` — validates commit messages in CI and via git hooks
- `commitizen` — interactive CLI prompt that builds conforming messages: `git cz` instead of `git commit`

### 3.2 git-cliff

git-cliff is a Rust-based changelog generator that parses conventional commits and renders them via a TOML-configured Tera template. It reads git history, applies tag-range filtering, and writes a `CHANGELOG.md` file. It does not make version bump commits or create tags — it only generates the changelog.

**Minimal `cliff.toml` config:**

```toml
[changelog]
header = "# Changelog\n\n"
body = """
{% if version %}\
## [{{ version | trim_start_matches(pat="v") }}] - {{ timestamp | date(format="%Y-%m-%d") }}
{% else %}\
## [Unreleased]
{% endif %}\
{% for group, commits in commits | group_by(attribute="group") %}
### {{ group | upper_first }}
{% for commit in commits %}
- {% if commit.scope %}**{{ commit.scope }}**: {% endif %}{{ commit.message }}\
{% endfor %}
{% endfor %}\n
"""
trim = true

[git]
tag_pattern = "v[0-9].*"
commit_parsers = [
  { message = "^feat", group = "Added" },
  { message = "^fix", group = "Fixed" },
  { message = "^refactor", group = "Changed" },
  { message = "^perf", group = "Performance" },
  { body = ".*BREAKING CHANGE", group = "Breaking Changes" },
  { message = ".*", skip = true },
]
```

**CLI usage:**

```bash
# Generate changelog for the upcoming 2.4.0 release (all commits since last tag)
git cliff --tag 2.4.0 -o CHANGELOG.md

# Append only the unreleased commits to an existing CHANGELOG.md
git cliff --unreleased --prepend CHANGELOG.md

# Preview changelog without writing to file
git cliff --unreleased
```

**Fit:** Works well in both Git Flow (run on the release branch before merge to `main`) and TBD (run as part of the CI release job before tagging). Because it only writes a file, it integrates into any pipeline without requiring special permissions or API tokens.

### 3.3 release-please

release-please is a Google-maintained GitHub Action and CLI that manages release PRs. It parses conventional commits merged to the target branch, determines the next version according to semver rules, bumps the version in a manifest file, updates `CHANGELOG.md`, and opens a PR titled `chore(main): release X.Y.Z`. The PR is kept up to date as more commits land; merging the PR is the human's only required action.

**Minimal GitHub Actions workflow:**

```yaml
# .github/workflows/release-please.yml
name: Release Please

on:
  push:
    branches: [main]

permissions:
  contents: write
  pull-requests: write

jobs:
  release-please:
    runs-on: ubuntu-latest
    steps:
      - uses: google-github-actions/release-please-action@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}
          release-type: node
          manifest-file: .release-please-manifest.json
```

**`.release-please-manifest.json`** (tracks current version per package):

```json
{
  ".": "2.3.0"
}
```

**Workflow:**

1. Commits land on `main` following conventional commits format.
2. release-please creates or updates a Release PR with the computed next version, bumped `package.json`, and `CHANGELOG.md` diff.
3. Team reviews and merges the PR when ready to ship.
4. The action tags the commit and creates the GitHub Release automatically.

**Fit:** Designed for TBD. Can be configured for Git Flow release branches but requires setting `target-branch` per release branch and managing multiple open PRs.

### 3.4 semantic-release

semantic-release is a Node.js CI release automation tool. On every push to the configured branch, it analyzes commits since the last tag, determines the next version, writes `CHANGELOG.md`, commits the changelog, creates the git tag, and publishes to npm and/or GitHub Releases — with no human step required after the triggering merge.

**Minimal `.releaserc.json`:**

```json
{
  "branches": ["main"],
  "plugins": [
    "@semantic-release/commit-analyzer",
    "@semantic-release/release-notes-generator",
    ["@semantic-release/changelog", {
      "changelogFile": "CHANGELOG.md"
    }],
    ["@semantic-release/git", {
      "assets": ["CHANGELOG.md", "package.json"],
      "message": "chore(release): ${nextRelease.version} [skip ci]\n\n${nextRelease.notes}"
    }],
    "@semantic-release/github"
  ]
}
```

**Plugin responsibilities:**

| Plugin | Role |
|--------|------|
| `@semantic-release/commit-analyzer` | Determines next version from conventional commits |
| `@semantic-release/release-notes-generator` | Builds the changelog body |
| `@semantic-release/changelog` | Writes/updates `CHANGELOG.md` |
| `@semantic-release/git` | Commits the updated files back to the branch |
| `@semantic-release/github` | Creates the GitHub Release |

**Fit:** Best for fully automated pipelines where no human review of the release PR is needed. Works on TBD trunk or on Git Flow's `main` branch after the release branch has been merged. Not suited to workflows where teams want to defer the release decision after merging code.

## 4. Comparison Table — Git Flow vs Trunk-Based Development

| Dimension | Git Flow | Trunk-Based Development |
|---|---|---|
| **Version bump timing** | Immediately on release branch cut, before stabilization | At tag time — manual on trunk, or automated by CI |
| **Version bump location (branch)** | `release/x.y.z` branch | `main` (trunk) — via direct commit or Release PR |
| **Version bump author** | Release manager (or CI triggered by branch creation) | Developer (manual) or fully automated CI (release-please / semantic-release) |
| **Changelog authoring moment** | After stabilization, before merge to `main` | When Release PR is merged (release-please) or immediately on merge to `main` (semantic-release) |
| **Changelog committed to** | Release branch → merges to `main` | `main` directly |
| **Tooling fit: git-cliff** | Natural fit — run on release branch before final merge | Works as changelog-only step in CI release job |
| **Tooling fit: release-please** | Possible with extra config (target-branch per release) | Native fit — designed for TBD trunk |
| **Tooling fit: semantic-release** | Runs on `main` post-merge; no release branch awareness | Native fit — best for fully automated TBD pipelines |
| **Human review gate before release** | Yes — release branch QA cycle | Optional — release-please provides PR gate; semantic-release skips it |
| **Rollback mechanism** | Revert commit on release branch before merge | Revert on trunk; requires re-tag |

## 5. Choosing Your Tooling Stack

Use this guide to select the right tooling for your workflow. All three tools require conventional commits as input; choose `commitlint` to enforce the discipline across the team.

| Decision | Recommended tool |
|----------|------------------|
| "I want a PR to review and approve before the release is cut" | **release-please** |
| "I want fully automated, zero-touch releases on every merge to main" | **semantic-release** |
| "I want to generate the changelog myself without automating the release" | **git-cliff** |
| "I'm using Git Flow with manual release management" | **git-cliff** + manual version bump commit on the release branch |
| "I want to enforce conventional commits in CI and locally" | **commitlint** (`@commitlint/config-conventional`) |

**Key constraint:** All three changelog/automation tools parse conventional commit messages. If your team does not write conventional commits, none of these tools will produce accurate output. Adopt commitlint first and enforce it for at least one release cycle before introducing release-please, semantic-release, or git-cliff.

**Typical stack combinations:**

- **Git Flow, manual releases:** commitlint + git-cliff (run on release branch) + manual version bump
- **TBD, team wants release approval gate:** commitlint + release-please
- **TBD, fully automated:** commitlint + semantic-release
- **TBD, changelog only (release managed separately):** commitlint + git-cliff in CI
